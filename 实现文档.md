# Vercel 在线网页小游戏实现文档

## 1. 项目目标

在 `vercel-mini-games` 目录下搭建一个可部署到 Vercel 的在线网页小游戏项目，首期包含以下 5 个游戏：

- `2048`
- `俄罗斯方块`
- `贪吃蛇`
- `蜘蛛纸牌`
- `扫雷`

本轮先输出实现文档，不直接开始编码。

## 2. 目标边界

### 2.1 本期要实现

- 一个游戏首页，展示 5 个游戏入口
- 每个游戏拥有独立页面路由
- 所有游戏均可在浏览器本地运行
- 支持桌面端为主的操作体验
- 支持基础的分数、重新开始、失败或胜利提示
- 支持本地持久化，如最高分、上次难度、音效开关等
- 项目可直接部署到 Vercel

### 2.2 本期暂不实现

- 账号系统
- 在线排行榜
- 多人联机
- 后端数据库
- 广告系统
- 原生 App 打包

## 3. 技术方案

### 3.1 方案选择

推荐使用：`Next.js + TypeScript + App Router + 原生 CSS Modules`

选择理由：

- `Next.js` 对 Vercel 部署最省事
- 纯前端小游戏不依赖服务端接口，部署成本低
- `TypeScript` 便于管理游戏状态和规则逻辑
- `CSS Modules` 足够应付本项目，不必额外引入复杂 UI 框架
- 后续如果要补首页 SEO、分享信息、主题切换，也容易扩展

### 3.2 运行模式

- 主要采用客户端渲染
- 游戏逻辑放在浏览器端执行
- 使用 `localStorage` 保存最高分、配置和进度
- 不使用数据库和 Serverless Function 作为首期前提

### 3.3 部署方式

- 代码仓库接入 Vercel
- 默认构建命令使用 `next build`
- 默认输出为 Next.js 项目
- 首期不需要单独配置 `vercel.json`

## 5. 页面与路由规划

### 5.1 首页

路由：`/`

功能：

- 展示项目标题与简介
- 展示 5 个游戏卡片
- 展示每个游戏的玩法简介
- 点击进入对应游戏页面

### 5.2 游戏页面

- `/games/2048`
- `/games/tetris`
- `/games/snake`
- `/games/spider`
- `/games/minesweeper`

每个页面统一包含：

- 游戏标题
- 开始或重开按钮
- 返回首页按钮
- 当前分数或状态栏
- 玩法提示
- 键盘操作提示
- 移动端兼容提示（如首版支持有限）

## 6. 公共模块设计

### 6.1 通用布局

建议封装 `GameShell`，统一处理：

- 页头
- 分数栏
- 操作区
- 状态提示
- 返回首页
- 游戏说明

### 6.2 本地存储

建议封装统一存储层，避免各游戏重复读写 `localStorage`：

- `bestScore`
- `lastMode`
- `soundEnabled`
- `theme`
- `resumeState`

### 6.3 输入处理

抽离统一输入工具，减少重复监听：

- 键盘方向键
- `WASD`
- 空格
- 回车
- 鼠标点击
- 扫雷右键

### 6.4 状态管理原则

- 每个游戏独立管理自己的状态
- 不强行引入全局状态库
- 首期优先使用 React 自带状态管理
- 游戏规则逻辑尽量放在 `lib/games/*` 下，避免全部写进页面组件

## 7. 各游戏实现方案

### 7.1 2048

核心功能：

- `4x4` 棋盘
- 方向移动
- 相同数字合并
- 随机生成新方块
- 分数统计
- 游戏结束判断
- 最高分保存

实现建议：

- 采用二维数组表示棋盘
- 每次移动拆成“压缩 -> 合并 -> 再压缩”三步
- 动画首期可先做基础过渡，不追求复杂特效

验收标准：

- 箭头键可操作
- 合并规则正确
- 失败检测正确
- 刷新后最高分仍在

### 7.2 俄罗斯方块

核心功能：

- 标准下落区域
- `7` 种方块
- 左右移动
- 旋转
- 加速下落
- 消行计分
- 游戏结束判断
- 下一块预览

建议边界：

- 首期先不做“暂存方块”
- 首期先不追求 T-Spin、连击等高级规则
- 对外展示时可使用“方块下落”作为页面副标题，降低商标风险

实现建议：

- 使用矩阵表示方块和地图
- 通过碰撞检测判断移动是否合法
- 以固定 tick 驱动自动下落
- 将速度提升与分数或消行数挂钩

验收标准：

- 方块生成、旋转、碰撞、落地、消行都正确
- 连续游玩不会明显卡顿
- 键盘操作响应稳定

### 7.3 贪吃蛇

核心功能：

- 网格地图
- 蛇移动
- 食物生成
- 吃到食物后增长
- 撞墙或撞自己失败
- 分数统计
- 重新开始

实现建议：

- 使用数组保存蛇身坐标
- 使用固定时间步推进移动
- 禁止同一帧直接反向移动
- 食物生成时避开蛇身

验收标准：

- 不会出现食物刷在蛇身上的情况
- 连续转向逻辑正常
- 失败条件判断稳定

### 7.4 蜘蛛纸牌

说明：

- 这是 5 个游戏里实现复杂度最高的一项
- 建议首期先做 `单花色 Spider`，确保规则和交互先跑通
- 后续再扩展到 `双花色` 或 `四色`

核心功能：

- 10 列牌堆布局
- 可拖动或点击移动牌组
- 仅允许合法序列移动
- 空列放牌规则
- 发牌堆补牌
- 完成同花顺序后自动收走
- 胜利判断

实现建议：

- 独立设计牌对象结构：花色、点数、朝向、唯一标识
- 明确区分“可见牌”和“不可见牌”
- 拖拽首期可先做点击选中加点击目标列，后续再补真实拖拽体验
- 先保证规则正确，再优化动画

验收标准：

- 牌序判断正确
- 发牌逻辑正确
- 自动收牌正确
- 至少单花色模式可完整通关

### 7.5 扫雷

核心功能：

- 难度选择
- 布雷
- 左键翻开
- 右键插旗
- 连片展开
- 首次点击不踩雷
- 胜利或失败判断
- 计时

实现建议：

- 使用二维格子结构记录是否有雷、是否翻开、是否插旗、周围雷数
- 将布雷延迟到第一次点击后生成，保证首击安全
- 连片展开使用 BFS 或 DFS

验收标准：

- 首击必安全
- 周围雷数计算正确
- 自动展开逻辑正确
- 插旗与胜利判断一致

## 8. UI 与交互建议

### 8.1 风格方向

- 首页采用轻量卡片式布局
- 游戏页保持统一顶部栏和操作栏
- 整体风格偏简洁，不先追求复杂视觉特效

### 8.2 响应式策略

- 首期优先适配桌面端
- 平板端尽量可用
- 手机端至少保证能打开页面和查看提示
- 对键盘强依赖的游戏，在手机端明确提示“建议桌面体验”

### 8.3 音效策略

- 首期可不加音效
- 若后续加入音效，默认关闭或允许手动关闭

## 9. 开发顺序建议

推荐按以下顺序实现：

1. 初始化 Next.js 项目与基础布局
2. 完成首页与公共 `GameShell`
3. 完成 `2048`
4. 完成 `贪吃蛇`
5. 完成 `扫雷`
6. 完成 `俄罗斯方块`
7. 完成 `蜘蛛纸牌` 单花色版
8. 补充样式细节、持久化和部署配置

这样安排的原因：

- `2048`、`贪吃蛇`、`扫雷` 规则边界清晰，适合先搭公共能力
- `俄罗斯方块` 对实时循环和碰撞要求更高，放后面更稳
- `蜘蛛纸牌` 复杂度最高，放最后可减少返工

## 11. 风险与注意事项

### 11.1 商标与素材风险

- `Tetris` 名称和经典视觉元素有商标风险
- 对外页面文案建议优先使用“俄罗斯方块玩法”或“方块下落”描述
- 不直接复用第三方受版权保护的素材、音效和贴图

### 11.2 蜘蛛纸牌复杂度高

- 若一开始就追求完整拖拽、撤销、发牌动画，开发量会明显增加
- 建议先做规则正确的 MVP

### 11.3 Next.js 客户端状态注意点

- `localStorage` 只能在客户端读取
- 游戏循环相关逻辑要注意组件卸载时清理定时器
- 键盘事件要避免重复绑定
